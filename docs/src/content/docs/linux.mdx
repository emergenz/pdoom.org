<Aside type="caution" title="Experimental">
  Linux support is **experimental** and evolving. What works depends heavily on
  your **display server** (X11 vs Wayland) and **compositor / desktop
  environment**. Read the [compatibility matrix](#compatibility-matrix) before
  relying on it, and see [known limitations](#known-limitations) for what isn't
  validated yet.
</Aside>

crowd-cast follows a strict **fail-closed** design: it never silently degrades.
If a prerequisite for correct capture isn't met in your environment, recording
is **blocked** with a clear reason rather than recording something incomplete or
wrong.

## Compatibility matrix

For each environment, this is what's currently implemented:

<div class="cap-matrix">

| Environment | App-window video | Follow-focus | System tray | Input |
| --- | --- | --- | --- | --- |
| **X11** (any WM) | <span class="dot ember">●</span> per-window | <span class="dot ember">●</span> yes | <span class="dot brass">●</span> SNI host | <span class="dot ember">●</span> rdev |
| **Wayland · wlroots** | <span class="dot dim">○</span> full-screen | <span class="dot ember">●</span> yes | <span class="dot brass">●</span> SNI host | <span class="dot ember">●</span> evdev |
| **Wayland · GNOME** | <span class="dot dim">○</span> full-screen | <span class="dot brass">●</span> extension | <span class="dot brass">●</span> AppIndicator | <span class="dot ember">●</span> evdev |
| **Wayland · KDE Plasma** | <span class="dot dim">○</span> full-screen | <span class="dot dim">○</span> blocked | <span class="dot ember">●</span> native | <span class="dot ember">●</span> evdev |
| **Wayland · other** | <span class="dot dim">○</span> full-screen | <span class="dot dim">○</span> blocked | <span class="dot brass">●</span> SNI host | <span class="dot ember">●</span> evdev |

</div>

**Legend:** <span class="dot ember">●</span> works · <span class="dot brass">●</span> works, with a dependency · <span class="dot dim">○</span> not available.

Notes:

- **wlroots** covers Sway, Hyprland, river, Wayfire and labwc.
- **Blocked** means recording is gated off until a focus provider exists; the agent fails closed rather than recording incorrectly.
- **System tray** needs a StatusNotifier host: KDE provides one natively, GNOME via the AppIndicator extension, and most bars (waybar, XFCE, MATE, LXQt) support it. **On Sway, run waybar; swaybar does not implement the protocol.**
- **evdev** input capture needs `input`-group membership; X11 uses `rdev` and needs no group change.
- **GNOME** follow-focus needs the bundled [extension](#gnome) and a relogin.
- **Not yet validated:** X11 per-app capture and the GNOME focus path are implemented and unit-tested, but not verified end-to-end on real hardware.

The short version: **X11** and **wlroots compositors (e.g. Sway/Hyprland) with
waybar** are the best-supported setups today. GNOME works for input gating only
after installing our shell extension. KDE and other Wayland compositors can show
the tray but recording is gated off until a focus provider lands.

## Requirements

- **x86_64** Linux. (Other architectures aren't packaged.)
- `tar` and `zstd` available (used to unpack the bundled capture components).
- On **Wayland**, membership in the **`input`** group (for `evdev` input
  capture). The installer sets this up for you.
- Screen capture (libobs) is **bundled**, so you don't need to install OBS.

## Installing

<Aside type="note">
  One-click packaged downloads aren't published for Linux yet. For now you
  install with the script in the repo, from a build of the agent.
</Aside>

The installer is **user-space**: it installs into `~/.local` and needs `sudo`
only once, to add you to the `input` group.

<Steps>

1. Build the agent and run the installer from a checkout:

   ```bash
   packaging/linux/install.sh --local
   ```

2. It lays out:

   ```text
   ~/.local/bin/crowd-cast-agent                   # the agent
   ~/.local/share/crowd-cast/obs/<abi>/...          # bundled capture components
   ~/.local/share/applications/crowd-cast.desktop   # menu entry
   ~/.config/autostart/crowd-cast.desktop           # start at login
   ```

3. It then adds you to the **`input`** group (the one `sudo` step). **Log out
   and back in** for that to take effect.

</Steps>

To remove it again (this keeps your config and `input`-group membership):

```bash
packaging/linux/install.sh --uninstall
```

## Display servers & per-app capture

- **X11**: crowd-cast can capture a **single window** (per-app) using
  XComposite, following the focused window. This needs a pure X11 session with
  `_NET_ACTIVE_WINDOW` and the Composite extension (standard on virtually all
  X11 desktops).
- **Wayland**: per-window video capture isn't available yet, so capture is
  full-screen. Follow-focus is still used to **gate input** (stop logging
  keystrokes/mouse when a non-selected app is focused).

## Follow-focus by environment

"Follow-focus" means knowing which app is in front, so input is only logged for
your selected apps. The mechanism differs per environment, and there is **no
fallback**: if no reliable provider exists for your environment, recording is
blocked:

- **X11**: reads the active window directly. Works.
- **wlroots** (Sway, Hyprland, river, Wayfire, labwc): uses the
  `wlr-foreign-toplevel` protocol. Validated on Sway.
- **GNOME**: see [below](#gnome); requires our extension.
- **KDE Plasma and other Wayland compositors**: not implemented yet, so
  recording is gated off.

### GNOME

GNOME (Wayland) provides **no app-callable focus API**, so crowd-cast ships a
small, read-only **GNOME Shell extension** that reports the focused window over
a private interface. Because Wayland can't hot-load a new extension, you'll need
to **install it and log out/in once** to activate it. Until it's live, recording
on GNOME is gated off. This path currently compiles but hasn't been validated on
a live GNOME session yet.

## Input capture

On **Wayland**, input is captured via **`evdev`**, which requires your user to be
in the **`input`** group. The installer adds you automatically; to do it by hand:

```bash
sudo usermod -aG input "$USER"
# then log out and back in
```

If you're not in the group, crowd-cast can't read input devices and will tell you
so (e.g. *"No input devices found. Make sure you are in the 'input' group."*). On
**X11**, input is captured via `rdev` and no group change is needed.

## System tray

The Linux tray uses the freedesktop **StatusNotifierItem (SNI)** protocol. The
icon and its menu only appear if your desktop runs a **StatusNotifier host**:

- **Works:** KDE Plasma (native), **waybar**, GNOME with the **AppIndicator**
  extension, XFCE, MATE, Cinnamon, LXQt.
- **Doesn't show the tray:** **swaybar** (Sway's default bar does *not* implement
  the SNI host protocol), and vanilla GNOME without the AppIndicator extension.

<Aside type="caution" title="Sway users">
  Under **Sway**, the default **swaybar** does not implement the tray protocol,
  so the crowd-cast menu won't open. Run **waybar** (with its `tray` module)
  instead and the icon and menu appear normally. The agent still runs and
  records even with no tray host; you just can't control it from the menu until
  a host is present.
</Aside>

## Autostart & updates

- **Start at login** is an XDG autostart entry at
  `~/.config/autostart/crowd-cast.desktop`.
- **Updates** are delivered as signed (Ed25519) releases that the agent verifies
  and installs in place, then relaunches, mirroring the Sparkle auto-update on
  macOS.

## Configuration

Your config lives at `~/.config/dev.crowd-cast.agent/config.toml`. The options
are the same across platforms. See [Configuration](/docs/crowd-cast/configuration/).

## Known limitations

- **Per-app (per-window) video capture is X11-only.** On Wayland, capture is
  full-screen.
- **X11 per-app capture is unit-tested but not yet live-validated.**
- **GNOME follow-focus** compiles but hasn't been exercised on a real GNOME
  session, and requires the bundled extension + a relogin.
- **KDE Plasma and other Wayland compositors** have no focus provider yet, so
  recording is gated off there.
- The tray won't appear under **swaybar** or vanilla GNOME (see above).

These and other platform quirks are tracked on the
[Known issues](/docs/crowd-cast/known-issues/) page.
